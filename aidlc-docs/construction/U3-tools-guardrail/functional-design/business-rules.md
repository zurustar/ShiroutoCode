# Business Rules — U3 Tools & Guardrail

> テスト可能な表明（TDD）。`[PBT]` は rapid 候補。**安全性の中核**＝過剰ブロック側に倒す（fail-safe）。

## ディスパッチ（単一窓口）

### R1. 全ツール実行は ToolDispatcher 経由（App Q5=A）
- 表明: Tool は Registry 経由でのみ取得され、実行は ToolDispatcher.Dispatch のみが行う（Agentが直接 Tool.Execute を呼ぶ経路は存在しない＝設計・レビューで担保）。
- 表明: Dispatch は必ず Evaluate を呼んでから実行可否を決める。

### R2. 判定→実行の対応（Q2=A）
- `Allow` → そのまま実行。
- `Confirm` → `Confirmer.Confirm` が true のときのみ実行、false なら未実行で「スキップ（拒否）」を返す。
- `Deny` → 実行せず理由を返す（Confirmerにも問わない）。
- `[PBT]` 任意の Decision で、実行が起きるのは (Allow) または (Confirm かつ確認true) のときに限る。

## スコープ限定（US-5.3, SECURITY-11）

### R3. ワークスペース実体パス封じ込め（Q3=A）
- 表明: File 系の対象パスは絶対化・`filepath.Clean`・**シンボリックリンク解決後**の実体が `WorkspaceRoot` 配下であること。
- 表明: 書込/削除がスコープ外 → `Deny`。読取がスコープ外 → `Confirm`。
- 表明: `..` やシンボリックリンクを使った脱出は実体解決により阻止される。
- `[PBT]` `WorkspaceRoot` 配下の任意の相対パスは内部判定で「内」。`../` を十分前置した任意パスは「外」。
- 表明（フェイルクローズ）: パス解決でエラー（不正パス等）→ 内と見なさない（Deny/Confirm）。
- 表明（F-02）: スコープ内であっても `.git/` 配下（特に `hooks/`）への書込/削除は `Confirm`。gitフック経由でコマンド denylist を迂回した任意コード実行に繋がるため。
- 表明（F-04 多層防御）: `FileTool` 自身も書込/削除時にワークスペース境界を再検証し、境界外なら（ディスパッチャを経由しない呼び出しでも）拒否する。
- 表明（F-04）: ワークスペース外の読取のうち、機微情報ストア（`~/.ssh`・`~/.aws`・`~/.gnupg`・`~/.netrc`・`/etc/shadow`・`id_rsa` 等）に該当するものは `Confirm` ではなく `Deny`。

## 危険コマンド denylist（US-5.2, Terminal）

### R4. Deny パターン（Q4=A）
次にマッチするコマンドは `Deny`:
- ルート/広域削除: `rm -rf /`, `rm -rf /*`, `rm -rf ~`, `rm` の絶対パスでワークスペース外
- デバイス/低レベル: 書込先 `/dev/...`, `dd of=/dev/...`, `mkfs`
- システム制御: `shutdown`, `reboot`, `halt`, `init 0/6`
- fork爆弾: `:(){ :|:& };:` 等のパターン
- パイプ実行: `curl ... | sh`（sh/bash/zsh/dash/ksh/fish に加え python/perl/ruby/node/php などインタプリタ全般を対象, F-01）
- 認証情報の外部送信が明白なもの（ベストエフォート）
- 表明（F-01）: 判定前に `${IFS}`/`$IFS` などの空白難読化を正規化し、トークン分割回避によるバイパスを防ぐ。

### R5. Confirm パターン（Q4=A）
- `sudo`、権限昇格、`chmod -R`/`chown -R` の広域、ワークスペース外への `mv`、ネットワーク到達が広いコマンド。
- 表明: denylist/confirmlist は設定（`ExtraDenyPatterns`）で追加可能。
- `[PBT]` 既知の各 Deny パターン文字列（バリエーション: 余分な空白/オプション順）に対し、判定は常に Deny（過剰側）。
- 表明（フェイルクローズ）: パース不能・判定不能なコマンドは Allowにしない（Confirm）。

## Git 危険操作（US-4.4/5.2）

### R6. Git 判定（Q5=A）
- Allow: `status`/`diff`/`log`/`add`/`commit`/`branch`/`checkout`/`switch`/`stash`/`pull(fast-forward)`。
- Confirm: `push`（リモート反映、既定確認）/`merge`/`rebase`。
- Deny もしくは Confirm（ConfirmModeに従う）: `push --force`/`-f`, `reset --hard`, `rebase`(履歴改変), `filter-branch`, `clean -fdx`, `commit --amend`+force push 連動。
- Confirm（F-07 ワークスペース外/hook 影響）: `git config --global`/`--system`、`core.hooksPath` の変更、`git -c <k>=<v>`（インライン設定上書き）、`--exec-path`。リポジトリローカルの `git config <key> <value>` は Allow。
- `[PBT]` force/hard/履歴改変フラグを含む git コマンドは Allow にならない。

## Web ツール（FR-4.4, US-4.5）

### R7. Web 制限（Q6=A）
- GET のみ。スキーム http/https のみ（file://, ftp:// 等は Deny）。
- 表明: ユーザー指示由来の取得のみ（エージェントが勝手に大量アクセスしない＝呼び出し回数はエージェントループ制御に従う）。
- 表明: レスポンスはサイズ上限（既定 例 1MiB）で打ち切り（メモリ保護）。
- 表明: 攻撃的（同一ホスト連打/ポート走査的パターン）は抑止対象（ベストエフォート、Deny/Confirm）。
- 表明（F-03 SSRF対策）: 名前解決後の宛先 IP を検査し、loopback・link-local（`169.254.0.0/16`＝クラウドメタデータ含む）・private（`10/8`,`172.16/12`,`192.168/16`）・unique-local IPv6・unspecified・multicast への接続を拒否する。検査は各リダイレクト先にも適用され、DNSリバインディングにも耐える。解決不能IPはフェイルクローズ（拒否）。

## 確認・フェイルクローズ（US-5.2/6.2, SECURITY-15）

### R8. 確認の伝達と非対話（Q7=A）
- 表明: `Confirm` 判定時、`Confirmer` が対話可能なら諾否を問う。**非対話（Confirmer未提供/非TTY）では確認できない → 実行しない（Deny相当）**。
- `[PBT]` 非対話 Confirmer 下では、Confirm/Deny いずれの判定でも実行は起きない。

### R9. フェイルクローズ（Q8=A）
- 表明: 評価中のエラー・未知の ActionKind・スコープ判定不能 → Allow にしない。
- 表明: ツール実行失敗時は失敗を明示し、危険側に進まない（中間状態を残さない努力: 例 編集は一時書込→rename 等）。

## 自動承認（US-5.1）

### R10. デフォルト自動承認
- 表明: 上記いずれの危険判定にも該当しないワークスペース内の通常操作は `Allow`（ユーザー確認なしで実行）。
- 表明: 実行された操作は結果（ToolResult）として記録され、事後に履歴で確認できる（U5表示）。

## 横断
- 全ツールの引数・パス・コマンドは検証（SECURITY-05）。判定ロジックは専用パッケージ `internal/guardrail` に分離（SECURITY-11）。ログはU1マスキング前提。

## セキュリティ前提・残存リスク（Threat Model / Trust Boundaries）

> 本ツールは「ローカル LLM が自律的にツールを呼ぶエージェント」であり、**LLM の出力は信頼できない入力**として扱う。ガードレール（`internal/guardrail`）が唯一の防御線である。以下は「**どんな仕様か → どんな使い方をすると → どんなリスクがありうるか → 利用者の対策**」を整理したもの。設計判断として現時点では受容している残存リスクを含む。

### TM-1. `run_command` は設計上「任意コマンド実行」である（既定 Allow 姿勢, F-01）
- **仕様**: `run_command` は `sh -c <command_line>` でユーザ権限の任意シェルコマンドを実行する。ガードレールはコマンドを **ブロックリスト方式** で評価し、R4/R5 の危険パターンに一致しないものは R10 により **確認なしで `Allow`**（非対話/CI でも同様）。
- **こういう使い方をすると**: 信頼できない内容（`web_fetch` で取得した Web ページ、読み込んだファイル、外部由来の課題文など）をプロンプトに含めて自律実行させる／無人（非対話・CI）で走らせる。
- **こんなリスク**: プロンプトインジェクションで誘導されたコマンドが、denylist の隙間（例: 変数間接化 `r=rm; $r ...`、`find / -delete`、`shred`、未対策のインタプリタ呼び出し、base64+eval など）を通って実行されうる。cwd はワークスペースだが、コマンド自体はユーザが到達できる全ファイル・全ネットワークにアクセスできる（封じ込めなし）。結果として機密読取・外部送信・破壊的操作が起こりうる。
- **設計判断（受容）**: denylist は「境界」ではなく**多層防御の一層（ベストエフォート）**。「未一致＝Allow」という既定姿勢は利便性とのトレードオフで現状維持。
- **利用者の対策 / 将来オプション**: 信頼できないコンテンツを扱う場合や無人運用では (a) `ExtraDenyPatterns` で対象環境固有の禁止を追加、(b) 対話モードで実行し確認を効かせる、(c) コンテナ/最小権限ユーザ/読み取り専用FS下で実行する。allow-list 方式・「対話時は全コマンド既定 `Confirm`」・`NonInteractivePolicy`（config 定義済みだが評価器未接続）の有効化は、製品方針決定後に導入予定。

### TM-2. 信頼境界＝ワークスペースとローカル LLM エンドポイント
- **仕様**: ファイル操作はワークスペースに封じ込め（R3）。LLM エンドポイントは既定 `http://localhost`。読み込んだファイル内容はそのままモデルへ送信される（エージェントの本質）。
- **こういう使い方をすると**: エンドポイントを**リモートの `http://`** に向ける／ワークスペースに機密ファイルを置いたまま実行する。
- **こんなリスク**: プロンプト・ファイル内容が平文ネットワークを流れ、盗聴・改ざんされうる（起動時に F-08 警告は出るが実行は止めない）。エンドポイントが信頼境界の外にあると、送信内容はその主体に渡る。
- **利用者の対策**: エンドポイントはローカルまたは信頼境界内に限定し、リモート利用時は `https` を用いる。ワークスペースに不要な機密を置かない。

### TM-3. `web_fetch` 取得物・ツール出力は「信頼できない入力」
- **仕様**: `web_fetch` は http(s) GET を行い（R7, SSRF対策あり）、本文をモデルに返す。同様にコマンド出力・ファイル内容もモデルに戻る。
- **こういう使い方をすると**: 外部 Web やリポジトリ内の任意テキストをエージェントに読ませる。
- **こんなリスク**: 取得物に「次にこのコマンドを実行せよ」等の**間接プロンプトインジェクション**が仕込まれ、後続のツール呼び出しを誘導されうる（TM-1 と連鎖）。SSRF 自体は F-03 で内部/メタデータIPを遮断済みだが、公開先からの応答内容の信頼性までは保証しない。
- **利用者の対策**: 取得元を限定し、TM-1 の対策（確認・隔離）を併用する。

### TM-4. シンボリックリンク TOCTOU（書き込み, F-05）
- **仕様**: 書込パスは「存在する最深祖先」のシンボリックリンクを解決して境界判定する（R3）。`FileTool` は書込直前にも境界を再検証する（F-04）。
- **こういう使い方をすると**: 他プロセス（別ユーザ/並行プロセス）が書き込み可能なワークスペースで実行する。
- **こんなリスク**: ガードレール判定〜実ファイル書込（`os.Rename`）の間に、親ディレクトリ等が**シンボリックリンクに差し替えられる**と、境界外へ書き込まれる競合（TOCTOU）が理論上残る。再検証で窓は縮小しているが完全には閉じていない。
- **設計判断（受容）**: 単一ユーザのローカル前提では費用対効果が低いため、`O_NOFOLLOW`/`openat` ベースの厳密なパス走査は未実装。
- **利用者の対策**: ワークスペースは実行ユーザ専有とし、他者書き込み可能な場所では実行しない。

### TM-5. 非対話/CI 実行時の確認不能
- **仕様**: `Confirm` 判定は対話 `Confirmer` が無い（非TTY/CI）と実行されない＝Deny 相当（R8）。確認読取エラー時も常に拒否（F-09）。
- **こういう使い方をすると**: CI 等で `Confirm` 相当の操作（sudo・push・`.git/` 書込・グローバル git 設定 等）を含むタスクを流す。
- **こんなリスク**: 当該操作はブロックされ**タスクが未完で停止**する（安全側だが運用上の制約）。一方 TM-1 の通り `Allow` 相当のコマンドは無人でも実行される点に注意。
- **利用者の対策**: 無人運用ではタスクを `Confirm` 不要な範囲に設計し、危険操作は人手の対話セッションで行う。

> 上記 TM-1〜TM-5 のうち、TM-1（既定 Allow 姿勢）と TM-4（TOCTOU 完全封じ込め）は**意図的に受容している残存リスク**。変更着手の判断材料は `aidlc-docs/security/security-audit-2026-06-10.md` を参照。
