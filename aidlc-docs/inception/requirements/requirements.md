## 1. Intent Analysis Summary

- **User Request（原文・初回）**: 「AI駆動開発を実現するツールを作りたい。利用者とのインタフェースはVSCode（プラグイン形式）。LLMはLMStudioで提供するローカルLLMにしたい。」
- **改定の経緯**: エージェントを **VSCode の外（ターミナル単体・他エディタ）でも利用する**ことが目標と確定。VSCode拡張はNode依存・配布制約があり、スタンドアロン配布は Go の単一静的バイナリが明確に有利。→ **Goでヘッドレスコア（CLI）を実装し、VSCode拡張は将来そのフロントとして薄く被せる**方針に転換。今回のMVPは **CLIのみ**。
- **Request Type**: New Project（グリーンフィールド）
- **Scope Estimate**: System-wide（CLI UI + エージェント実行エンジン + LLM連携 + ツール実行層 + ガードレール）
- **Complexity Estimate**: Complex（自律エージェント、複数ファイル編集、ターミナル実行、安全制御）
- **License**: GPL-3.0（copyleft互換）
- **User-facing docs language**: 日本語（README基準）

## 2. Product Vision

ローカルLLM（LM Studio）を頭脳とする**自律型AIコーディングエージェント**を開発する。中核は **Go製のヘッドレスエンジン**で、**CLI（ターミナル）から対話的に操作**する。利用者が自然言語で指示を与えると、エージェントが多段階に推論（plan→act→observe）し、ワークスペース内のファイル編集・ターミナルコマンド実行・Git操作・Web情報取得を**基本的に自動で**実行してタスクを完遂する。破壊的・悪意ある操作はセーフティガードレールで防止する。

エンジンはフロントエンドから独立しているため、**将来 VSCode拡張やその他エディタを“薄いフロント”として被せられる**。VSCode拡張フロントは**今回のスコープ外（後続フェーズ）**。

## 3. Functional Requirements

> 注: 「UIに表示/記録」は本MVPでは **CLI（ターミナル）出力** を指す。

### FR-1: CLIインタフェース（対話型）
- ターミナルで動作する**対話型CLI**を提供する。ユーザーの指示入力、エージェントの応答・思考過程・実行アクションの履歴を時系列で表示する。
- 実行中のツール呼び出し（ファイル編集・コマンド実行等）と結果を逐次出力する。
- 単発実行（`shiroutocode "指示"`）と対話セッション（REPL）の双方を想定（詳細は設計段階）。
- **[将来]** 同じエンジンに VSCode拡張等の別フロントを被せられるよう、UI非依存のコアAPIを保つ。

### FR-2: ローカルLLM連携（LM Studio）
- LM Studio が提供する **OpenAI互換 REST API**（`/v1/chat/completions`、既定 `http://localhost:1234/v1`）を直接呼び出す。
- プロバイダは **LM Studio固定**（マルチプロバイダ抽象は持たない）。
- ストリーミング応答（SSE）に対応し、生成テキストを逐次CLIへ反映する（Goの `net/http` + SSEパース）。
- **[Assumption A1]** エンドポイントURL/ポートおよび使用モデル名は**設定（設定ファイル / フラグ / 環境変数）から変更可能**とする。プロバイダ追加の抽象化は行わない。

### FR-3: 自律エージェントループ
- ユーザー指示を起点に、LLM出力に基づきツールを選択・実行し、結果を観測して次の行動を決める **多段階エージェントループ**（plan→act→observe）を実装する。
- 1指示で**複数ファイルにまたがる自動編集**を完遂できること（MVPゴール）。
- ループの終了条件（タスク完了判定、最大ステップ数、ユーザー中断）を持つ。

### FR-4: ツール（エージェントが実行できる操作）
- **FR-4.1 ファイル読み取り**: ワークスペース内ファイルの読み取り。
- **FR-4.2 ファイル変更**: ワークスペース内ファイルの新規作成・編集・削除。
- **FR-4.3 ターミナル実行**: ビルド・テスト等のコマンド実行（Goの `os/exec`、出力キャプチャ）。
- **FR-4.4 Webアクセス**: 外部情報の取得。
- **FR-4.5 Git操作**: コミット・ブランチ作成等。
- すべてのツール実行はCLIに記録・可視化される。

### FR-5: 承認フロー & セーフティガードレール
- **デフォルトは自動承認**: 通常のファイル変更・コマンド実行はユーザー承認なしで自動実行する。
- **セーフティガードレール（ブロック対象）**: 以下は自動実行を**拒否**し、人間の明示的な確認を必須とする（または実行不可）:
  - ワークスペース外への破壊的操作（例: ワークスペース外の削除、`rm -rf /` 等のシステム破壊）
  - 外部システムへの攻撃的・悪意ある操作（スキャン/攻撃/DoS等）
  - 取り消し困難・高影響の操作（例: 強制push、履歴改変、認証情報の外部送信）
- **[Assumption A2]** ガードレールは「ワークスペースへのファイル操作スコープ限定」＋「危険コマンドのdenylist/パターン検出」＋「該当時は自動承認をバイパスして明示確認を要求」で実現。承認時の確認は**CLIプロンプト（対話）**で行う。細部は設計段階で確定。

### FR-6: 設定
- LM Studioエンドポイント、モデル名、最大ステップ数、ガードレール挙動などを**設定ファイル（例: プロジェクト配下/ホーム配下）＋コマンドラインフラグ＋環境変数**で構成可能にする（優先順位は設計段階で確定）。

## 4. Non-Functional Requirements

### NFR-1: プラットフォーム & 技術スタック
- 中核エンジン＋CLIを **Go** で実装（単一静的バイナリで配布、ランタイム依存なし、クロスプラットフォーム）。
- 配布形態: スタンドアロンバイナリ（GitHub Releases等）。`go install` 対応も想定。
- **[将来]** VSCode拡張フロント（TypeScript）は後続フェーズ。コアとは別プロセス（IPC: stdio上のJSON-RPC等を想定）で接続する設計余地を残す。

### NFR-2: 完全ローカル動作 / プライバシー
- LLM推論はローカル（LM Studio）で完結し、コード等の機微情報を外部送信しない（FR-4.4のWebアクセスはユーザー指示に基づく明示的なもののみ）。

### NFR-3: 性能 / 応答性
- ストリーミングによりLLM応答を逐次表示し、体感応答性を確保する。
- エージェントループの長時間実行中も入力（中断）に応答できる（Goのgoroutine + context によるキャンセル）。

### NFR-4: 信頼性 / 安全な失敗
- ツール実行・LLM呼び出し・ファイルI/Oは明示的なエラーハンドリングを持ち、エラー時は**フェイルクローズ**（危険側に倒れない）こと（SECURITY-15）。
- LM Studio未起動・接続失敗時は分かりやすいエラーをCLIに表示する。

### NFR-5: 可観測性 / ログ
- 構造化ログ（タイムスタンプ、相関ID、ログレベル、メッセージ）を持ち、機微情報（トークン/PII）をログに出さない（SECURITY-03）。

### NFR-6: テスト容易性
- ビジネスロジック（エージェントループ、ガードレール判定、API変換層）は単体テスト可能な構造とする。
- **プロパティベーステスト（PBT）を適用**（拡張有効, PBT-09: Go → **rapid**）。

### NFR-7: 保守性 / セキュア設計
- セキュリティ上重要なロジック（ガードレール、コマンド検証）を専用パッケージに分離する（SECURITY-11）。
- 依存はロックファイル（`go.sum`）で固定し脆弱性スキャン（`govulncheck` 等）を行う（SECURITY-10）。

### 5.1 Security 適用範囲の暫定整理
ローカルで動作するCLIツールであり、クラウド/Webサーバ/データストアを持たないため、多くのSECURITYルールはN/A想定。関連が高いもの:
- **SECURITY-05（入力検証）**: LLM出力・ツール引数の検証（特にコマンド/パス）。
- **SECURITY-09（ハードニング/誤設定防止）**: デフォルト認証情報なし、エラー時に内部情報を露出しない。
- **SECURITY-10（サプライチェーン）**: `go.sum`固定、`govulncheck`、依存最小化。
- **SECURITY-11（セキュア設計）**: ガードレール=ビジネスロジック濫用（misuse case）対策の中核。
- **SECURITY-13（完全性）**: LLM出力/外部データの安全な取り扱い（unsafeなデシリアライズ回避）。
- **SECURITY-15（例外処理/フェイルセーフ）**: フェイルクローズ、トップレベルエラーハンドリング。
- N/A想定: SECURITY-01/02/04/06/07/08/12/14（クラウド・Webサーバ・ユーザ認証前提）。最終判定は各段階で実施。

## 6. Assumptions（要レビュー）
- **A1**: プロバイダはLM Studio固定だが、エンドポイントURL/ポートとモデル名は設定変更可能。
- **A2**: ガードレールは「ワークスペース限定スコープ + 危険コマンドdenylist + 該当時のCLI明示確認」で実現。細部は設計段階で確定。
- **A3**: CLIコアはフロント非依存に設計し、将来のVSCode拡張フロントはIPC経由の別プロセスとして接続する（今回は未実装）。

## 7. Out of Scope（現時点）
- **VSCode拡張フロントエンド（今回はCLIのみ。VSCodeフロントは後続フェーズ）**。
- 複数LLMプロバイダ（OpenAI/Anthropic等）の切替。
- クラウド/サーバサイドコンポーネント、マルチユーザー認証。
- インライン補完。

## 8. Key Requirements Summary
ローカルLLM（LM Studio, OpenAI互換REST）を頭脳とし、**Go製のヘッドレスエンジン＋対話型CLI**から、ファイル/ターミナル/Git/Webの各ツールを**自動承認で**駆使してマルチファイル編集タスクを完遂する自律エージェント。破壊的・悪意ある操作はセーフティガードレールで防止。単一静的バイナリで配布し、VSCodeの外でも利用可能。将来は同一コアにVSCode拡張等の薄いフロントを被せる。Security/PBT(rapid)拡張をBlockingで適用。
