# Requirements — ShiroutoCode

## 1. Intent Analysis Summary

- **User Request（原文）**: 「AI駆動開発を実現するツールを作りたい。利用者とのインタフェースはVSCode（プラグイン形式）。LLMはLMStudioで提供するローカルLLMにしたい。」
- **Request Type**: New Project（グリーンフィールド／ゼロから新規開発）
- **Scope Estimate**: System-wide（VSCode拡張UI + エージェント実行エンジン + LLM連携 + ツール実行層）
- **Complexity Estimate**: Complex（自律エージェント、複数ファイル編集、ターミナル実行、安全制御を含む）
- **License**: GPL-3.0（copyleft互換のソースであること）
- **User-facing docs language**: 日本語（README基準）

## 2. Product Vision

VSCode上で動作する**自律型AIコーディングエージェント拡張**を開発する。利用者がサイドバーのチャットUI（Claude Code的）から指示を与えると、エージェントがローカルLLM（LM Studio経由）を用いて多段階に推論し、ワークスペース内のファイル編集・ターミナルコマンド実行・Git操作・Web情報取得を**基本的に自動で**実行してタスクを完遂する。破壊的・悪意ある操作はセーフティガードレールで防止する。最終的にVS Code Marketplace等での一般公開を目指す。

## 3. Functional Requirements

### FR-1: チャットUI（サイドバーWebview）
- VSCodeのサイドバーにチャットパネル（Webview）を提供する（Q4=A）。
- ユーザーの指示入力、エージェントの応答・思考過程・実行したアクションの履歴を時系列で表示する。
- 実行中のツール呼び出し（ファイル編集・コマンド実行等）と結果をUI上に可視化する。

### FR-2: ローカルLLM連携（LM Studio）
- LM Studio が提供する **OpenAI互換 REST API**（`/v1/chat/completions`、既定 `http://localhost:1234/v1`）を直接呼び出す（Q5=A）。
- プロバイダは **LM Studio固定**（マルチプロバイダ抽象は持たない）（Q6=B）。
- ストリーミング応答（SSE）に対応し、生成テキストを逐次UIへ反映する。
- **[Assumption A1]** 一般公開（Q9=C）のため、エンドポイントURL/ポートおよび使用モデル名はVSCode設定（settings）から変更可能とする。プロバイダ追加の抽象化は行わない。

### FR-3: 自律エージェントループ
- ユーザー指示を起点に、LLMの出力に基づきツールを選択・実行し、結果を観測して次の行動を決める **多段階エージェントループ**（plan→act→observe）を実装する（Q1=B, Q8=C）。
- 1指示で**複数ファイルにまたがる自動編集**を完遂できること（MVPゴール, Q8=C）。
- ループの終了条件（タスク完了判定、最大ステップ数、ユーザー中断）を持つ。

### FR-4: ツール（エージェントが実行できる操作）（Q2=A,B,C,D,E）
- **FR-4.1 ファイル読み取り**: ワークスペース内ファイルの読み取り。
- **FR-4.2 ファイル変更**: ワークスペース内ファイルの新規作成・編集・削除。
- **FR-4.3 ターミナル実行**: ビルド・テスト等のコマンド実行。
- **FR-4.4 Webアクセス**: 外部情報の取得。
- **FR-4.5 Git操作**: コミット・ブランチ作成等。
- すべてのツール実行はUI上に記録・可視化される。

### FR-5: 承認フロー & セーフティガードレール（Q3）
- **デフォルトは自動承認**: 通常のファイル変更・コマンド実行はユーザー承認なしで自動実行する。
- **セーフティガードレール（ブロック対象）**: 以下は自動実行を**拒否**し、人間の明示的な確認を必須とする（または実行不可とする）:
  - ワークスペース外への破壊的操作（例: ワークスペース外の削除、`rm -rf /` 等のシステム破壊）
  - 外部システムへの攻撃的・悪意ある操作（スキャン/攻撃/DoS等）
  - 取り消し困難・高影響の操作（例: 強制push、履歴改変、認証情報の外部送信）
- **[Assumption A2]** ガードレールは「ワークスペースへのファイル操作スコープ限定」＋「危険コマンドのdenylist/パターン検出」＋「該当時は自動承認をバイパスして明示確認を要求」で実現する。承認ポリシーの細部は設計段階（Functional/NFR Design）で確定する。

### FR-6: 設定
- LM Studioエンドポイント、モデル名、最大ステップ数、ガードレール挙動などをVSCode設定で構成可能にする。

## 4. Non-Functional Requirements

### NFR-1: プラットフォーム & 技術スタック
- VSCode拡張として実装。実装言語は **TypeScript**（Q7=A）。
- VSCode Extension API / Webview API を使用。
- 配布形態: VS Code Marketplace等での一般公開（Q9=C）→ パッケージング（vsce）とライセンス（GPL-3.0）整合が必要。

### NFR-2: 完全ローカル動作 / プライバシー
- LLM推論はローカル（LM Studio）で完結し、コード等の機微情報を外部送信しない（FR-4.4のWebアクセスはユーザー指示に基づく明示的なもののみ）。

### NFR-3: 性能 / 応答性
- ストリーミングによりLLM応答を逐次表示し、UIの体感応答性を確保する。
- エージェントループの長時間実行中もVSCode UIをブロックしない（非同期実行・キャンセル可能）。

### NFR-4: 信頼性 / 安全な失敗
- ツール実行・LLM呼び出し・ファイルI/Oは明示的なエラーハンドリングを持ち、エラー時は**フェイルクローズ**（危険側に倒れない）こと（SECURITY-15）。
- LM Studio未起動・接続失敗時は分かりやすいエラーをUIに表示する。

### NFR-5: 可観測性 / ログ
- 構造化ログ（タイムスタンプ、相関ID、ログレベル、メッセージ）を持ち、機微情報（トークン/PII）をログに出さない（SECURITY-03）。

### NFR-6: テスト容易性
- ビジネスロジック（エージェントループ、ガードレール判定、API変換層）は単体テスト可能な構造とする。
- **プロパティベーステスト（PBT）を適用**（拡張有効, PBT-09: TypeScript → fast-check）。

### NFR-7: 保守性 / セキュア設計
- セキュリティ上重要なロジック（ガードレール、コマンド検証）を専用モジュールに分離する（SECURITY-11）。
- 依存はロックファイルで固定し脆弱性スキャンを行う（SECURITY-10）。

### 5.1 Security 適用範囲の暫定整理（詳細は設計/コード段階で評価）
本プロジェクトはローカルで動作するVSCode拡張であり、クラウド/Webサーバ/データストアを持たないため、多くのSECURITYルールはN/Aの想定。要件段階で特に関連が高いもの:
- **SECURITY-05（入力検証）**: LLM出力・ツール引数の検証（特にコマンド/パス）。
- **SECURITY-09（ハードニング/誤設定防止）**: デフォルト認証情報なし、エラー時に内部情報を露出しない。
- **SECURITY-10（サプライチェーン）**: lockファイル、依存脆弱性スキャン、`latest`タグ不使用。
- **SECURITY-11（セキュア設計）**: ガードレール=ビジネスロジック濫用（misuse case）対策の中核。
- **SECURITY-13（完全性）**: LLM出力/外部データの安全な取り扱い（unsafeなdeserialization回避）。
- **SECURITY-15（例外処理/フェイルセーフ）**: フェイルクローズ、グローバルエラーハンドラ。
- N/A想定: SECURITY-01/02/04/06/07/08/12（一部）/14（クラウド・Webサーバ・ユーザ認証前提のもの）。最終判定は各段階で実施。

## 6. Assumptions（要レビュー）
- **A1**: プロバイダはLM Studio固定だが、公開配布のためエンドポイントURL/ポートとモデル名は設定変更可能とする。
- **A2**: ガードレールは「ワークスペース限定スコープ + 危険コマンドdenylist + 該当時の明示確認バイパス」で実現。細部は設計段階で確定。

## 7. Out of Scope（現時点）
- 複数LLMプロバイダ（OpenAI/Anthropic等）の切替（将来拡張の余地は残すが今回は対象外）。
- クラウド/サーバサイドコンポーネント、マルチユーザー認証。
- インライン補完（Q4でAのみ選択。B/C/Dは現時点スコープ外）。

## 8. Key Requirements Summary
ローカルLLM（LM Studio, OpenAI互換REST）を頭脳とし、TypeScript製VSCode拡張のサイドバーチャットUIから、ファイル/ターミナル/Git/Webの各ツールを**自動承認で**駆使してマルチファイル編集タスクを完遂する自律エージェント。破壊的・悪意ある操作はセーフティガードレールで防止。Security/PBT拡張をBlockingで適用し、最終的にMarketplace公開を目指す。
