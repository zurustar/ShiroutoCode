# User Stories — ShiroutoCode

> **改定履歴**: 2026-06-08 アーキテクチャ転換に伴い、UIを **VSCodeサイドバー → CLI（ターミナル）** に変更。E1=CLIインタフェース、E2=LM Studio接続・設定（設定ファイル/フラグ/環境変数）。旧版は `stories.md.backup.20260608T000000Z`。VSCode拡張フロントのストーリーは後続フェーズで追加。

- **分割アプローチ**: Epic-Based（E1〜E6）
- **ペルソナ**: P1 開発者「タクミ」（[personas.md](personas.md)）。全ストーリーの主体。
- **受け入れ基準**: 主要シナリオは Given-When-Then、補助条件はチェックリスト併用。
- **INVEST**: 各ストーリーは Independent / Negotiable / Valuable / Estimable / Small / Testable を満たすよう記述。
- 対応する要件IDは requirements.md（FR-/NFR-）を参照。

## E1: CLIインタフェース（対話型）

### US-1.1 CLIから指示を入力する
**As** 開発者P1, **I want** ターミナルのCLIから自然言語で指示を入力したい, **so that** エディタやシェルを離れずにAIへ作業を依頼できる。
（FR-1）

**Acceptance Criteria (Given-When-Then):**
- **Given** `shiroutocode` をワークスペースで起動した **When** 指示を引数（`shiroutocode "..."`）または対話プロンプトに入力する **Then** 指示が受理され、エージェント処理が開始される。
- **Given** 対話セッション（REPL） **When** 指示を入力して送信する **Then** 入力内容が会話履歴として表示され、処理が始まる。

**Checklist:**
- [ ] 空入力は処理を開始しない
- [ ] 単発実行（引数）と対話セッション（REPL）の両方をサポート
- [ ] 端末がカラー非対応/非TTYでも壊れない（プレーン出力にフォールバック）

---

### US-1.2 会話・思考・アクション履歴を時系列で確認する
**As** 開発者P1, **I want** エージェントの応答・思考過程・実行アクションを時系列で見たい, **so that** 何が起きているか把握できる。
（FR-1, FR-4）

**Acceptance Criteria (Given-When-Then):**
- **Given** エージェントが処理中 **When** LLM応答やツール実行が発生する **Then** それぞれが区別可能な形（メッセージ/ツール呼び出し/結果）でターミナルに逐次出力される。

**Checklist:**
- [ ] ユーザー発話・AI応答・ツール実行・結果が視覚的に区別される（ラベル/色）
- [ ] 長い出力は省略表示し、必要に応じ全文確認手段を持つ
- [ ] ストリーミング中もハングせず逐次表示される

---

### US-1.3 実行を中断する
**As** 開発者P1, **I want** 実行中のエージェント処理を中断したい, **so that** 意図と違う動きを止められる。
（FR-3, NFR-3）

**Acceptance Criteria (Given-When-Then):**
- **Given** エージェントループが実行中 **When** 中断操作（Ctrl-C 等）を行う **Then** 進行中のステップが安全に停止し、中断済みである旨が表示される。

**Checklist:**
- [ ] 中断後はワークスペースが不整合な中間状態で放置されない（可能な範囲で安全に停止）
- [ ] 中断は数秒以内に反映される（context キャンセルで伝播）

---

## E2: LM Studio接続・設定

### US-2.1 エンドポイント・モデルを設定する
**As** 開発者P1, **I want** LM StudioのエンドポイントURL/ポートとモデル名を設定（設定ファイル/フラグ/環境変数）で指定したい, **so that** 自分の環境のローカルLLMに接続できる。
（FR-2, FR-6, Assumption A1）

**Acceptance Criteria (Given-When-Then):**
- **Given** 設定でエンドポイント（既定 `http://localhost:1234/v1`）とモデル名を指定 **When** 指示を実行する **Then** 指定先・指定モデルへリクエストが送られる。

**Checklist:**
- [ ] 未設定時は既定値を用いる
- [ ] 不正なURL形式は起動時または送信前に検証される（SECURITY-05）
- [ ] プロバイダはLM Studio固定（他プロバイダ抽象は持たない）
- [ ] 設定の優先順位（フラグ > 環境変数 > 設定ファイル > 既定）が一貫している

---

### US-2.2 応答をストリーミングで受け取る
**As** 開発者P1, **I want** LLMの応答が逐次（ストリーミング）表示されてほしい, **so that** 待ち時間の体感を減らせる。
（FR-2, NFR-3）

**Acceptance Criteria (Given-When-Then):**
- **Given** LM Studioがストリーミング応答に対応 **When** エージェントがLLMを呼ぶ **Then** 生成テキストがトークン単位で逐次ターミナルに反映される。

**Checklist:**
- [ ] ストリーミング中も中断に応答できる（非ブロッキング）
- [ ] ストリーム中断・完了・エラーが正しく扱われる

---

## E3: 自律エージェントループ

### US-3.1 指示一つで複数ファイルを自動編集して完遂する
**As** 開発者P1, **I want** 一つの指示で複数ファイルにまたがる変更を自動で完遂してほしい, **so that** 手作業を減らして任せられる（**MVPの中核**）。
（FR-3）

**Acceptance Criteria (Given-When-Then):**
- **Given** 複数ファイルの変更を要する指示（例: 「関数名Xを全体でYにリネームし、テストも更新」） **When** 実行する **Then** エージェントが必要なファイルを特定・編集し、完了時に変更点の要約を提示する。

**Checklist:**
- [ ] plan→act→observe の反復で進行する
- [ ] 変更したファイル一覧が結果に提示される
- [ ] タスク完了 or 中断 or 最大ステップ到達で必ず終了する

---

### US-3.2 ループの進行状況が見える
**As** 開発者P1, **I want** エージェントが今どのステップにいるか（計画/実行/観測）を見たい, **so that** 進捗と意図を理解できる。
（FR-3）

**Acceptance Criteria (Given-When-Then):**
- **Given** エージェントループ実行中 **When** 各ステップに遷移する **Then** 現在のステップと実行アクションがターミナルに表示される。

**Checklist:**
- [ ] 現在のステップ番号 / 最大ステップ数が分かる

---

### US-3.3 安全に終了する（暴走防止）
**As** 開発者P1, **I want** ループが無限に続かないようにしてほしい, **so that** 暴走やリソース浪費を防げる。
（FR-3, NFR-4）

**Acceptance Criteria (Given-When-Then):**
- **Given** 最大ステップ数が設定されている **When** 完了判定に達せずステップ上限に到達する **Then** ループを停止し、未完了である旨を表示する。

**Checklist:**
- [ ] 最大ステップ数は設定で変更可能（FR-6）
- [ ] 完了判定の根拠が結果に示される

---

## E4: ツール実行と可視化

### US-4.1 ファイルを読み取る
**As** 開発者P1, **I want** エージェントがワークスペース内ファイルを読み取れるようにしたい, **so that** 文脈に基づいた変更ができる。（FR-4.1）

**Acceptance Criteria (Given-When-Then):**
- **Given** 指示が既存コードの理解を要する **When** エージェントがファイル読み取りツールを使う **Then** 読み取り対象パスがターミナルに記録される。

**Checklist:**
- [ ] 読み取りはワークスペース配下に限定される（FR-5, SECURITY-05）

---

### US-4.2 ファイルを作成・編集・削除する
**As** 開発者P1, **I want** エージェントがワークスペース内のファイルを作成・編集・削除できるようにしたい, **so that** 変更タスクを完遂できる。（FR-4.2）

**Acceptance Criteria (Given-When-Then):**
- **Given** 変更が必要 **When** エージェントがファイル変更ツールを使う **Then** 変更内容（作成/編集/削除）と対象パスがターミナルに記録される。

**Checklist:**
- [ ] 変更はワークスペース配下に限定（外部はガードレールでブロック, E5）
- [ ] 削除は破壊的操作としてガードレール評価の対象（US-5.2）

---

### US-4.3 ターミナルコマンドを実行する
**As** 開発者P1, **I want** エージェントがビルド/テスト等のコマンドを実行できるようにしたい, **so that** 変更の検証まで任せられる。（FR-4.3）

**Acceptance Criteria (Given-When-Then):**
- **Given** 検証が必要な指示 **When** エージェントがコマンド実行ツール（`os/exec`）を使う **Then** 実行コマンドと出力（終了コード含む）がターミナルに記録される。

**Checklist:**
- [ ] 危険なコマンドはガードレール評価の対象（US-5.2, SECURITY-05/11）
- [ ] 実行は中断可能（US-1.3）

---

### US-4.4 Git操作を行う
**As** 開発者P1, **I want** エージェントがコミットやブランチ作成などのGit操作を行えるようにしたい, **so that** 変更を記録・整理できる。（FR-4.5）

**Acceptance Criteria (Given-When-Then):**
- **Given** Git管理下のワークスペース **When** エージェントがGitツールを使う **Then** 実行したGit操作がターミナルに記録される。

**Checklist:**
- [ ] 取り消し困難な操作（強制push・履歴改変等）はガードレール評価の対象（US-5.2）

---

### US-4.5 Webから情報を取得する
**As** 開発者P1, **I want** エージェントが必要に応じてWebから情報を取得できるようにしたい, **so that** 最新の情報を反映できる。（FR-4.4, NFR-2）

**Acceptance Criteria (Given-When-Then):**
- **Given** 外部情報が必要な指示 **When** エージェントがWebアクセスツールを使う **Then** アクセス先URLがターミナルに記録される。

**Checklist:**
- [ ] Webアクセスはユーザー指示に基づく明示的なもののみ（NFR-2）
- [ ] 攻撃的アクセス（スキャン/DoS等）はガードレールでブロック（US-5.2）

---

## E5: 承認フロー & セーフティガードレール

### US-5.1 通常操作は自動承認で実行される
**As** 開発者P1, **I want** 通常のファイル変更・コマンド実行を都度承認なしで自動実行してほしい, **so that** 中断されずスムーズに作業が進む。（FR-5）

**Acceptance Criteria (Given-When-Then):**
- **Given** 操作がガードレールの危険判定に該当しない **When** エージェントがその操作を行う **Then** ユーザー承認を求めずに実行され、結果がターミナルに記録される。

**Checklist:**
- [ ] 自動承認がデフォルト挙動である
- [ ] 実行された操作は事後に履歴で確認できる（US-1.2）

---

### US-5.2 危険・破壊的・攻撃的操作はブロックし明示確認を求める
**As** 開発者P1, **I want** システム破壊や外部攻撃などの危険操作は自動実行せずブロックしてほしい, **so that** 取り返しのつかない被害を防げる。（FR-5, SECURITY-11/15）

**Acceptance Criteria (Given-When-Then):**
- **Given** エージェントがガードレール上「危険」と判定される操作（例: ワークスペース外の削除、`rm -rf /`、強制push、外部への攻撃的アクセス、認証情報の外部送信）を試みる **When** その実行直前 **Then** 自動実行はバイパスされ、操作内容を提示して**CLIで明示的な確認**を求める（または実行不可とする）。
- **Given** 危険操作の確認が提示される **When** ユーザーが拒否する **Then** その操作は実行されずスキップされる。

**Checklist:**
- [ ] ガードレール判定はコマンド/パス/操作種別のパターンで行う（Assumption A2）
- [ ] 判定ロジックは専用パッケージに分離される（SECURITY-11）
- [ ] ブロック/確認要求はターミナルに明示される
- [ ] フェイルクローズ: 判定不能時は危険側に倒さず確認を求める（SECURITY-15）
- [ ] 非対話実行時（非TTY/CI）は危険操作を確認できないため安全側で停止する

---

### US-5.3 操作はワークスペースにスコープ限定される
**As** 開発者P1, **I want** ファイル操作が原則ワークスペース配下に限定されるようにしたい, **so that** 環境全体への影響を避けられる。（FR-5, Assumption A2）

**Acceptance Criteria (Given-When-Then):**
- **Given** エージェントがワークスペース外パスへの書き込み/削除を試みる **When** その実行直前 **Then** ガードレールがブロックし明示確認を求める（US-5.2に従う）。

**Checklist:**
- [ ] ワークスペースルートの判定が正しく行われる
- [ ] シンボリックリンク等によるスコープ回避を考慮する（パス正規化）

---

## E6: エラー処理・接続失敗時のUX

### US-6.1 LM Studio未起動・接続失敗時に分かりやすく案内する
**As** 開発者P1, **I want** LM Studioに接続できないとき原因と対処が分かるメッセージを見たい, **so that** すぐに復旧できる。（FR-2, NFR-4）

**Acceptance Criteria (Given-When-Then):**
- **Given** LM Studioが未起動またはエンドポイントが誤っている **When** 指示を実行する **Then** 「接続できません」と原因・確認ポイント（起動状態/URL/モデル）を示すメッセージがターミナルに表示される。

**Checklist:**
- [ ] 内部スタックトレースや機微情報を露出しない（SECURITY-09）
- [ ] 設定の確認方法（該当フラグ/環境変数/設定ファイル）を示す

---

### US-6.2 ツール実行失敗時にフェイルクローズして状況を表示する
**As** 開発者P1, **I want** ツール実行が失敗したとき安全側で停止し、状況を表示してほしい, **so that** 壊れた状態のまま進まない。（NFR-4, SECURITY-15）

**Acceptance Criteria (Given-When-Then):**
- **Given** ファイルI/Oやコマンド実行が失敗 **When** エラーが発生する **Then** エージェントは危険側に進まず、失敗内容を提示して停止または再判断する。

**Checklist:**
- [ ] 全外部呼び出し（LLM/ファイル/コマンド/Web）に明示的エラーハンドリングがある
- [ ] エラー時にリソースが適切に解放される（接続/ハンドル、defer）
- [ ] ユーザー向けエラーは一般化メッセージ（内部情報を出さない）
